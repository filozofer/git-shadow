# 🎨 git-shadow — Design System

> Code for thinking ≠ code for collaboration

---

## 🧠 Philosophy

git-shadow is built on a fundamental idea:

- 🟢 **visible code** → collaboration
- 🟣 **shadow code** → thinking

This duality is reflected in every visual element.

---

## 🎯 Core Concept

> Two representations of the same code:
>
> - a **public version** (clean, shared)
> - a **shadow version** (local, cognitive)

## 🧱 Logo

### Structure

```

[icon] git shadow

````

### Icon rules

- Represents a **Git branch**
- Includes a **shadow offset**
- Two layers:
  - main branch → 🟢
  - shadow branch → 🟣

### Principles

- minimal
- geometric
- no gradients
- no 3D
- no visual noise

## 💻 CLI Identity

git-shadow lives in the terminal.

### Color usage

- `git` → green
- `shadow` → purple

Example:

```bash
git shadow feature start my-feature
````

* 1200 × 630
* dark background

## 🚫 Avoid

* gradients
* shadows (CSS-style blur)
* skeuomorphic effects
* overly complex icons

---

## 🚀 Design Principles

* **Clarity over decoration**
* **Meaning over aesthetics**
* **Duality is key**
* **Think like a developer, not a designer**
